💼 LinkedIn Profile Header — The Signature Banner-and-Avatar Overlap: A Full-Bleed Cover Photo, a Circular Avatar Pulled Up Over Its Bottom Edge With a White Ring and z-index, an Editable Name / Headline / Location, Photos You Swap From Disk via Object URLs, an #OPEN TO WORK Ring, and a Header That Restacks on Mobile

Edit your name and headline in place, drop in a cover photo and an avatar, flip to a phone width — the header that fronts every profile on the web.

The profile header is one of the most copied layouts online — LinkedIn, X, Facebook, GitHub, Notion all wear the same hat: a full-bleed banner with a circular avatar pulled up over its bottom edge. The whole trick is one negative margin plus a z-index, and a white ring (a 4px border) that cuts the avatar cleanly out of the photo behind it. Both images use object-fit: cover so any picture fills its frame without stretching. Hit Edit to change the name, headline and location right on the card (they are contenteditable), tap the camera chips to swap the banner and avatar from your own files (read straight into an object URL), toggle Open to for the green #OPEN TO WORK ring, and press Mobile to watch the header restack. No library.

🔨 Build it yourself

    
A profile header looks like a lot — a cover photo, a floating round avatar with a crisp white ring, editable text, swappable images, a green open-to-work badge, a layout that folds onto a phone — but underneath it is a handful of honest CSS tricks and about a hundred lines of vanilla JS. The overlap that makes the whole thing recognisable is one negative margin plus a z-index; the clean edge is a 4px white border; photos never stretch because of object-fit: cover. Editing is contenteditable text synced to a read-out; swapping a photo is a hidden <input type="file"> read into a object URL; and going mobile is a flex row that wraps plus a couple of shrunk sizes. No framework, no dependencies.

    
    
      
  1. 
        
    1
    The markup: a banner, then a body with the avatar first
    
        
    The structure mirrors what you see: a .banner box holding a cover <img>, then a .pbody that starts with the .avatar-wrap and continues with the name, headline, meta and action buttons. The avatar lives inside the body, not the banner — that is what lets a negative margin pull it up while everything below flows normally underneath it. The whole card is a labelled role="region".
    
        
    <div class="pcard" role="region" aria-label="Profile header">
  <div class="banner"><img id="bannerImg" alt="Profile banner"></div>
  <div class="pbody">
    <div class="avatar-wrap">
      <img class="avatar" id="avatarImg" alt="Profile photo">
    </div>
    <h3 class="pname">Alex Rivera</h3>
    <p  class="pheadline">Senior Product Designer…</p>
    <div class="pactions">…Open to · Add section · More…</div>
  </div>
</div>
    
      
    2. 
      
  2. 
        
    2
    The overlap: one negative margin + a z-index
    
        
    This is the signature move. Give the avatar a fixed size (say 128px) and a negative margin-top a little more than half its height — -72px — so it climbs up and straddles the banner's bottom edge, roughly half over the photo and half over the white body. A z-index guarantees it paints above the banner. Because the avatar sits in normal flow, the name and everything after it simply start below where the avatar ends — no absolute positioning, no magic numbers to keep two layers in sync. (Absolute positioning is the alternative, but the negative-margin version reflows for free.)
    
        
    .banner      { height:140px; }              /* the photo strip   */
.avatar-wrap { width:128px; height:128px;   /* fixed circle      */
               margin-top:-72px;            /* <- pull UP over it */
               position:relative; z-index:2;/* <- paint on top    */ }
/* half the avatar (64px) + a little = 72px of overlap */

  3. 
        
    3
    The white ring: a border cuts the avatar out of the photo
    
        
    Sitting a round photo directly on a busy banner looks muddy. A 4px white border on the avatar (plus a soft shadow) reads as a clean ring that lifts the face off whatever is behind it — the same device Twitter and GitHub use. Because border-radius:50% makes the element a circle, the border becomes a ring automatically. Swap the ring colour to green and you have LinkedIn's #OPEN TO WORK state for free.
    
        
    .avatar {
  border-radius:50%;
  border:4px solid #fff;                 /* the ring */
  box-shadow:0 2px 8px rgba(15,23,42,.18);
  object-fit:cover;                       /* see step 4 */
}
.avatar-wrap.otw .avatar { border-color:#01754f; }  /* open to work */
    
      
    4. 
      
  4. 
        
    4
    object-fit: cover — real photos, no stretch
    
        
    People upload photos of every shape. If you let an <img> fill a fixed box it will squash. object-fit: cover tells the image to fill its frame while keeping its aspect ratio, cropping the overflow — exactly like background-size: cover but on a real <img> (so it stays accessible and has alt text). Set it on both the banner and the avatar and any picture, portrait or landscape, looks intentional.
    
        
    .banner img { width:100%; height:100%; object-fit:cover; }
.avatar     { width:100%; height:100%; object-fit:cover; }
/* fill the frame, keep aspect ratio, crop the excess —
   a tall selfie and a wide landscape both look right */

  5. 
        
    5
    Edit in place: contenteditable behind a toggle
    
        
    Rather than a separate form, the name, headline and location are the same elements you read — they just become editable. One Edit button flips a boolean, toggles contenteditable on those three nodes, and adds an .editing class to the card so CSS can reveal a dashed outline and the camera chips. Press it again and the outlines vanish. The text you see is the text you edit; there is no second copy to keep in sync.
    
        
    let editing = false;
const fields = [pname, pheadline, ploc];

editBtn.addEventListener("click", () => {
  editing = !editing;
  pcard.classList.toggle("editing", editing);
  fields.forEach(el => el.contentEditable = editing);   // same nodes, now editable
  editBtn.textContent = editing ? "✓ Done" : "✎ Edit profile";
  if (editing) pname.focus();
  render();
});
    
      
    6. 
      
  6. 
        
    6
    Live sync: type → read-out updates instantly
    
        
    Because the fields are real DOM nodes, an input listener fires on every keystroke. Read their textContent, and update the side panel — name length, headline length, whether a location is set. This is the same "the screen is a function of state" idea as any live form: one render() derives every displayed number from the current text, so nothing can drift out of date.
    
        
    [pname, pheadline, ploc].forEach(el =>
  el.addEventListener("input", render));

function render(){
  roName.textContent = pname.textContent.trim().length;
  roHead.textContent = pheadline.textContent.trim().length;
  roLoc.textContent  = ploc.textContent.trim() ? "set" : "empty";
  roMode.textContent = editing ? "Editing" : "Viewing";
}

  7. 
        
    7
    Swap a photo: a file input → an object URL
    
        
    A camera chip is just a button that clicks a hidden <input type="file" accept="image/*">. When a file is chosen, URL.createObjectURL(file) hands you a tiny blob: URL that points straight at the picked file in memory — no upload, no base64 — and you drop it into the <img>'s src. Always revokeObjectURL the previous one so the browser can free it. (FileReader.readAsDataURL is the other route; it gives a self-contained data URL but a much larger string.)
    
        
    bannerCam.addEventListener("click", () => bannerFile.click());

let lastBanner = null;
bannerFile.addEventListener("change", e => {
  const file = e.target.files[0]; if (!file) return;
  if (lastBanner) URL.revokeObjectURL(lastBanner);   // free the old blob
  lastBanner = URL.createObjectURL(file);            // blob:… object URL
  bannerImg.src = lastBanner;
  bannerImg.style.display = "block";
  render();
});

  8. 
        
    8
    Responsive: let the buttons wrap, shrink the avatar
    
        
    Going mobile needs no new markup. The action row is a flex container, so on a narrow card it wraps; a @media query (mirrored here by an .is-mobile class so you can preview it with a button) shrinks the banner and avatar, drops the negative margin to match, scales the name down, and switches the buttons to a full-width vertical stack. Same DOM, two layouts — the header just folds.
    
        
    .pactions { display:flex; flex-wrap:wrap; gap:8px; }   /* wraps when tight */

@media (max-width:560px){
  .banner       { height:100px; }
  .avatar-wrap  { width:88px; height:88px; margin-top:-48px; } /* smaller + less pull */
  .pname        { font-size:1.2rem; }
  .pactions     { flex-direction:column; align-items:stretch; }
  .pbtn         { width:100%; justify-content:center; }        /* full-width stack */
}
    
      
    9. 
      
  9. 
        
    9
    The green ring + accessibility finish
    
        
    The Open to button toggles an .otw class that turns the ring green and reveals a #OPEN TO WORK pill, mirrored to aria-pressed. Every image carries real alt text; the camera controls are focusable buttons with labels; and a visually-hidden aria-live region announces each change ("Banner updated.", "Open to work: on.") so a screen-reader user follows along. That is the whole header — a negative margin, a white ring, object-fit, editable text, an object URL, and a flex row that wraps.
    
        
    openBtn.addEventListener("click", () => {
  const on = avatarWrap.classList.toggle("otw");
  openBtn.setAttribute("aria-pressed", on);
  openBtn.classList.toggle("on", on);
  announce("Open to work: " + (on ? "on" : "off") + ".");
  render();
});
function announce(msg){ live.textContent = msg; }   // aria-live region
