🃏 Swipe Cards — A Tinder-Style Deck: Drag & Rotate a Card, a Threshold + Velocity Fling-to-Dismiss, an Undo Stack & a Keyboard-and-Screen-Reader-Accessible Gesture

Grab the top card. Drag it. Fling it left or right — or use the buttons and the arrow keys.

A working swipe deck built from scratch in vanilla JS — the gesture that made Tinder, and now lives in every "yes / no / skip" flow. One idea makes it work: the top card follows your pointer, and its whole personality — how far it slides, how much it rotates, whether a green LIKE or red NOPE stamp fades in — is just arithmetic on how far you have dragged from where you grabbed it. Let go past a threshold (or with enough velocity) and it flings off-screen and the next card is promoted; let go short and it springs back. Drag it (mouse or touch), tap Nope / Like, press ← → to swipe or U to undo. No library.

🔨 Build it yourself

    
A swipe card feels physical — you grab it, it tilts, it flies away — but underneath it is one number and a scrap of arithmetic. A stack of absolutely-positioned cards sits in a small box; only the top one listens to the pointer. Every move records dx = how far you have dragged sideways, and that single value drives the slide (translateX), the tilt (rotate), and how strongly a LIKE or NOPE stamp shows. On release you make one decision — past a distance threshold or thrown with enough velocity it flings off and the next card is promoted; otherwise it springs back. One pointer handler covers mouse, touch and pen; buttons and the keyboard call the exact same code; and a history stack makes undo free. ~110 lines of vanilla JS, no dependencies.

    
    
      
  1. 
        
    1
    The markup: a deck of stacked cards
    
        
    The deck is one positioned box; each card is position:absolute; inset:0 so they all pile up in the same spot. The top card carries the LIKE and NOPE stamps that will fade in during a drag. The whole deck is a focusable role="group" with an aria-label spelling out the gesture, and a visually-hidden aria-live region will announce each decision to a screen reader.
    
        
    <div id="deck" role="group" tabindex="0"
     aria-label="Swipe deck. ←/→ to pass/like, U to undo.">
  <!-- cards are injected here, top card last-painted / highest z -->
  <div class="card top">
    <div class="card-photo">
      <span class="stamp like">LIKE</span>
      <span class="stamp nope">NOPE</span>
    </div>
    <div class="card-info"> … name · role · tags … </div>
  </div>
</div>
<div class="sr-only" role="status" aria-live="polite"></div>

  2. 
        
    2
    The stack: absolute cards + depth transforms
    
        
    Because every card is inset:0, they overlap perfectly. To make the deck read as a deck, the cards behind the top one are pushed down and scaled down a touch — depth 1 gets a small translateY + scale(.95), depth 2 a bit more — and anything deeper than a few is hidden for performance. touch-action:none tells the browser "I'll handle the gesture, don't scroll", and will-change:transform hints the compositor.
    
        
    .deck  { position:relative; width:300px; height:420px; touch-action:none; }
.card  { position:absolute; inset:0; border-radius:20px;
         touch-action:none; user-select:none; will-change:transform; }
.card.animate { transition: transform .45s cubic-bezier(.22,.85,.28,1),
                            opacity .45s ease; }

/* set per-card in JS from its depth in the stack */
function stackStyle(el, depth){
  el.style.zIndex    = 100 - depth;
  el.style.transform = `translateY(${depth*14}px) scale(${1 - depth*0.05})`;
  el.style.opacity   = depth > 3 ? 0 : 1;
}
    
      
    3. 
      
  3. 
        
    3
    Track the drag: pointerdown / move with capture
    
        
    On pointerdown over the top card, remember where the grab started and call setPointerCapture so every later move and the release are delivered here even if the pointer leaves the card — which it will, mid-fling. Drop the animate class while dragging so the card tracks the finger with zero lag. Each pointermove computes dx / dy — the delta from the grab point — which is the only geometry the whole widget needs.
    
        
    let drag = null;
deck.addEventListener("pointerdown", e => {
  const el = topCard(); if (!el || !el.contains(e.target)) return;
  el.classList.remove("animate");          // follow the finger, no easing
  el.setPointerCapture(e.pointerId);
  drag = { x0:e.clientX, y0:e.clientY, id:e.pointerId,
           lastX:e.clientX, lastT:performance.now(), vx:0 };
});
deck.addEventListener("pointermove", e => {
  if (!drag || e.pointerId !== drag.id) return;
  const dx = e.clientX - drag.x0, dy = e.clientY - drag.y0;
  const now = performance.now(), dt = now - drag.lastT;
  if (dt) drag.vx = (e.clientX - drag.lastX) / dt;   // px per ms
  drag.lastX = e.clientX; drag.lastT = now;
  render(dx, dy);
});
    
      
    4. 
      
  4. 
        
    4
    The live transform: translate + rotate from dx
    
        
    This is the trick that sells it. The card is translated by the raw drag delta, and it rotates in proportion to how far it has moved sideways — a small factor like dx * 0.05, clamped so it never spins past ~18°. That tiny coupling of horizontal distance to tilt is the entire "physical card" illusion; drag right and it leans right, drag left and it leans left, exactly like a real card pinched at one corner.
    
        
    function render(dx, dy){
  const el = topCard();
  const rot = clamp(dx * 0.05, -18, 18);     // tilt ∝ horizontal drag
  el.style.transform =
    `translate(${dx}px, ${dy}px) rotate(${rot}deg)`;
  setStamps(el, dx);                         // step 5
}
const clamp = (v, lo, hi) => Math.max(lo, Math.min(hi, v));
    
      
    5. 
      
  5. 
        
    5
    Direction feedback: fade the LIKE / NOPE stamps
    
        
    The user needs to know which way counts as yes before they commit. Map the same dx to stamp opacity: drag right reveals LIKE (dx / threshold, clamped 0–1), drag left reveals NOPE (-dx / threshold). At the threshold the stamp is fully opaque, so "the stamp is solid" and "letting go will commit" become the same visual signal — the interface teaches its own rule.
    
        
    const THRESHOLD = 100;   // px to count as a decision
function setStamps(el, dx){
  el.querySelector(".stamp.like").style.opacity =
    clamp(dx / THRESHOLD, 0, 1);
  el.querySelector(".stamp.nope").style.opacity =
    clamp(-dx / THRESHOLD, 0, 1);
}
    
      
    6. 
      
  6. 
        
    6
    Release: threshold + velocity decide
    
        
    On pointerup, put the animate class back so whatever happens next eases smoothly, then make one decision. Commit if the card was dragged past the distance threshold or flicked with enough speed — a fast flick should count even if it did not travel far, which is what makes the gesture feel snappy rather than laborious. The sign of the drag (falling back to the sign of the velocity for a pure flick) picks like vs nope.
    
        
    deck.addEventListener("pointerup", e => {
  if (!drag || e.pointerId !== drag.id) return;
  const el = topCard();
  el.classList.add("animate");
  const dx = e.clientX - drag.x0, vx = drag.vx;
  drag = null;
  const dir = dx !== 0 ? Math.sign(dx) : Math.sign(vx);
  if (Math.abs(dx) > THRESHOLD || Math.abs(vx) > 0.6)
    commit(dir > 0 ? "like" : "nope");     // step 7
  else
    springBack(el);                        // step 8
});
    
      
    7. 
      
  7. 
        
    7
    Fling it off + promote the next card
    
        
    Committing pops the top card out of the model, then transitions it fully off-screen (well past the deck width, with a bit of extra rotation) and fades it out. Push a record onto a history stack for undo. Re-laying the remaining cards is what promotes the next one to the top — and because those cards carry the animate class, they glide up into place for free. Clean up the flung node on transitionend, with a timeout fallback.
    
        
    function commit(decision){
  const card = cards.shift();               // remove from the model
  const el = card.el, dir = decision === "like" ? 1 : -1;
  history.push({ data: card.data, decision });   // for undo
  el.style.transform =
    `translate(${dir*(deck.offsetWidth+220)}px,-40px) rotate(${dir*22}deg)`;
  el.style.opacity = 0;
  const done = () => el.remove();
  el.addEventListener("transitionend", done, { once:true });
  setTimeout(done, 500);                    // fallback if no transition
  relayStack();                             // next card becomes top
  announce(`${decision==="like"?"Liked":"Passed on"} ${card.data.name}.`);
}
    
      
    8. 
      
  8. 
        
    8
    Spring back under threshold
    
        
    If the release did not qualify, the card is not a decision — it must return to dead-centre. With the animate class on, that is a one-liner: set the transform back to zero and clear the stamps, and the CSS transition eases it home. This "no harm done" recovery is a huge part of why the gesture feels safe to try — a tentative half-drag costs nothing.
    
        
    function springBack(el){
  el.style.transform = "translate(0,0) rotate(0deg)";
  setStamps(el, 0);          // fade both stamps out as it returns
}

    9
    Buttons + keyboard: the same code path
    
        
    A drag-only card is unusable by keyboard and hostile on desktop, so the Nope / Like buttons and the arrow keys must drive the identical commit() the gesture uses — ← passes, → likes, U undoes. Routing every input through one function means the animation, the history push and the announcement are automatic for buttons and keys, with no duplicated logic. Only preventDefault the keys you actually handle.
    
        
    btnNope.onclick = () => topCard() && commit("nope");
btnLike.onclick = () => topCard() && commit("like");
btnUndo.onclick = undo;

document.addEventListener("keydown", e => {
  if (look.classList.contains("hidden")) return;   // only on this tab
  if (e.key === "ArrowLeft")  topCard() && commit("nope");
  else if (e.key === "ArrowRight") topCard() && commit("like");
  else if (e.key.toLowerCase() === "u") undo();
  else return;
  e.preventDefault();
});

    10
    Undo with a history stack + live-region a11y
    
        
    Because every commit pushed { data, decision }, undo is just "pop and put it back": rebuild the card, start it off-screen on the side it left from, unshift it as the new top, then relay the stack so it slides home. Finally, accessibility is not an afterthought — each decision and undo writes a sentence to the aria-live="polite" region so a screen-reader user hears "Liked Aria." or "Brought back Marco." exactly when a sighted user sees the card move.
    
        
    function undo(){
  if (!history.length) return;
  const { data, decision } = history.pop();
  const el = makeCard(data), dir = decision === "like" ? 1 : -1;
  el.classList.add("animate");
  el.style.transform =
    `translate(${dir*(deck.offsetWidth+220)}px,-40px) rotate(${dir*22}deg)`;
  el.style.opacity = 0;
  deck.appendChild(el);
  cards.unshift({ el, data });
  requestAnimationFrame(() => requestAnimationFrame(relayStack)); // paint, then glide in
  announce(`Brought back ${data.name}.`);
}
function announce(msg){ live.textContent = msg; }
