🎯 Stepper — A Multi-Step Wizard With Numbered Progress, Per-Step Validation Gating, Linear vs Non-Linear Jumps, a State Machine & A11y

Walk a long form through one step at a time.

A working 4-step sign-up wizard built from scratch in vanilla JS — Account → Profile → Payment → Review. The header tracks a real state machine: each step is upcoming, active, or done ✓. Next advances only if the current step validates — bad fields show inline errors and block you. Back returns to the previous step, keeping everything you typed. Flip non-linear on and you can click any completed step to jump straight back to edit it. The last step reviews every value before Submit. No library.

🔨 Build it yourself

    
A stepper is a small state machine: an array of steps, a current index, a done[] flag per step, and one data object that survives the whole flow. The header is a pure projection of that state; Next is gated by a per-step validator; Back and jump-to-completed are just index moves constrained by whether the target step is reachable. ~120 lines of vanilla JS, no dependencies.

    
    
      
  1. 
        
    1
    Model the state, not the DOM
    
        
    Describe the flow as data first. A STEPS array holds each step's label and description; current is the active index; done[] records which steps have passed validation; one data object collects every field so nothing is scoped to a single panel. Everything the UI shows is derived from these four values — render is a function of state.
    
        
    const STEPS = [
  { key:"account", label:"Account", desc:"Login details" },
  { key:"profile", label:"Profile", desc:"About you" },
  { key:"payment", label:"Payment", desc:"Billing" },
  { key:"review",  label:"Review",  desc:"Confirm" },
];
let current = 0;                       // active step index
const done  = [false,false,false,false];  // has each step validated?
const data  = {};                      // one bag for all steps' values

    2
    Derive each step's state, render the header
    
        
    Every circle is in exactly one of three states, and the rule is tiny: the active index is active, anything flagged done is done ✓, the rest are upcoming. The connector bar before step i fills when step i-1 is done. Re-run this projection after every state change — never poke individual nodes.
    
        
    function stepState(i){
  if (i === current) return "active";
  if (done[i])       return "done";
  return "upcoming";
}
STEPS.forEach((s,i) => {
  circle[i].className = "step-item " + stepState(i);
  if (i > 0) bar[i].classList.toggle("fill", done[i-1]);  // connector
  circle[i].querySelector(".num").textContent =
    stepState(i) === "done" ? "✓" : (i+1);
});

    Show one panel at a time
    
        
    Keep all panels in the DOM and toggle visibility — do not destroy and rebuild them, because the fields (and the values in them) must survive when the user steps back and forth. Hiding rather than removing is what makes persistence free.
    
        
    document.querySelectorAll(".wizstep").forEach(p => {
  p.classList.toggle("hidden", +p.dataset.step !== current);
});
// Back/Next just move `current`; the inputs are never re-created,
// so whatever the user typed is still sitting in the DOM.

    4
    Gate Next behind per-step validation
    
        
    This is the heart of a wizard. Each step owns a validator that returns an errors map — empty means valid. Next runs it first: if there are errors, paint them inline and do not move; only a clean pass sets done[current]=true and advances. Forward motion is impossible while the current step is invalid.
    
        
    const RULES = {
  0: v => ({
    email: /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(v.email) ? "" : "Enter a valid email.",
    password: (v.password||"").length >= 8 ? "" : "Min 8 characters.",
  }),
  /* 1: profile, 2: payment … */
};
function validate(i){
  const errs = RULES[i] ? RULES[i](readFields(i)) : {};
  paintErrors(errs);
  return Object.values(errs).every(m => m === "");   // true = clean
}
function next(){
  const ok = validate(current);
  done[current] = ok;
  if (!ok) return;                 // blocked — errors are showing
  if (current < STEPS.length-1) current++;
  render();
}

    Back + jump-to-completed (linear vs non-linear)
    
        
    Back is unconditional — going backward can never land you on invalid data. Jumping forward is the interesting policy: in linear mode you may only reach a step through Next, so circles are inert; in non-linear mode you may click any already-done step to jump straight to it. The guard is one predicate.
    
        
    function canJump(i){                       // is a header click allowed?
  if (i === current) return false;
  return nonLinear && done[i];             // only completed steps, only non-linear
}
function goTo(i){ if (canJump(i)){ current = i; render(); } }
function back(){ if (current > 0){ current--; render(); } }
// linear mode: canJump is always false → the header is display-only.

    Persist data + build the review step
    
        
    Because panels are only hidden, the inputs remember themselves — but you still want a single source of truth for the summary and eventual submit. Snapshot fields into data as each step passes, then the review step is just a read-out of data. The confirm screen is where the user catches their own mistakes before anything is committed.
    
        
    function readFields(i){ /* {email, password} for step 0, etc. */ }
// on a clean validate, fold the step's fields into the shared bag:
Object.assign(data, readFields(current));

function renderReview(){
  reviewBox.innerHTML = [
    row("Email", data.email), row("Name", data.name),
    row("Country", data.country), row("Card", mask(data.card)),
  ].join("");
}

    Progress indication + the Submit swap
    
        
    Give the user a sense of distance: a Step X of N label, a percentage, and a fill bar — all derived from current. On the last step the primary button becomes Submit, and Back is disabled on the first step so there is nowhere invalid to go.
    
        
    const total = STEPS.length, isLast = current === total - 1;
countEl.textContent = `Step ${current+1} of ${total}`;
const pct = Math.round(current / (total - 1) * 100);   // review = 100%
progFill.style.width = pct + "%";
nextBtn.textContent  = isLast ? "Submit ✓" : "Next →";
backBtn.disabled     = current === 0;
nextBtn.onclick = () => isLast ? submit() : next();

    Keyboard + ARIA
    
        
    Expose the machine to assistive tech. Mark the active circle with aria-current="step"; when a step is blocked, set aria-invalid on the bad fields and move focus to the first one so the error is announced and reachable. Clickable circles in non-linear mode must be real buttons (Enter/Space + focus ring). Never rely on colour alone — the ✓ glyph and label carry the state too.
    
        
    circle[i].setAttribute("aria-current", i === current ? "step" : "false");

function paintErrors(errs){
  let firstBad = null;
  for (const id in errs){
    const el = field(id), msg = errs[id];
    el.classList.toggle("invalid", !!msg);
    el.setAttribute("aria-invalid", msg ? "true" : "false");
    errEl(id).textContent = msg;
    if (msg && !firstBad) firstBad = el;
  }
  if (firstBad) firstBad.focus();     // land the user on the problem
}
