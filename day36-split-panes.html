🪟 Split / Resizable Panes — A Draggable Gutter: Pointer-Math Resize, Min/Max Constraints, Keyboard-Resizable & the ARIA separator Pattern

Grab the divider. Drag it. Or focus it and use the arrow keys.

A working split view built from scratch in vanilla JS — the same widget that splits your code editor from its preview, a mail list from a message, or a file tree from a document. One idea makes it work: a draggable gutter sits between two panes, and every drag becomes a single number — what percent of the container does the first pane get? — which you clamp to a min/max and write straight to one flex-basis. Drag the gutter (mouse or touch), double-click it to reset, flip the orientation, or click the divider and press ← → / ↑ ↓ to resize by keyboard. The right pane itself splits again — nesting is free. No library.

Dragging is just arithmetic (pointer position → percent → clamp), the keyboard is the WAI-ARIA separator pattern, and nesting a split inside a pane needs no new code at all.

🔨 Build it yourself

    
A split pane looks physical — you grab a bar and shove it — but underneath it is one number and a little arithmetic. Two panes sit in a flex container with a gutter between them; the first pane is pinned to a flex-basis percentage and the second one soaks up the rest. Dragging turns a pointer position into that percentage, clamped to a min and max. One pointer-event handler covers mouse, touch and pen at once; the same apply(percent) function is driven by the keyboard for the accessible separator pattern; and because every rule is scoped to a single .split, dropping a split inside a pane just works. ~90 lines of vanilla JS, no dependencies.

    
    
      
  1. 
        
    1
    The markup: two panes with a separator between them
    
        
    The structure is three boxes in a row: a pane, a gutter, another pane. The gutter is the interactive part, so it carries the accessibility contract up front — role="separator", tabindex="0" so it can be focused, and the aria-value* trio that describes its position as a number from min to max. That is the entire DOM; everything after this just moves one boundary.
    
        
    <div class="split is-row">
  <div class="pane" id="paneA"> … </div>

  <div class="gutter" role="separator" tabindex="0"
       aria-orientation="vertical" aria-controls="paneA"
       aria-valuemin="20" aria-valuemax="80" aria-valuenow="45"></div>

  <div class="pane"> … </div>
</div>
    
      
    2. 
      
  2. 
        
    2
    The layout: one pane gets a basis, the other fills
    
        
    Flexbox does the heavy lifting. The container is a flex row (or column). The primary pane — the first child — is fixed to a flex-basis that JavaScript will keep rewriting; the second pane is flex:1 1 0 so it always absorbs whatever is left after the primary and the gutter. Resizing therefore never touches the second pane at all: you only ever write one number, the primary's basis, and flex re-flows the rest. min-width:0 stops long content from refusing to shrink.
    
        
    .split           { display:flex; height:100%; }
.split.is-row    { flex-direction:row; }
.split.is-col    { flex-direction:column; }

.split > .pane   { flex:1 1 0; min-width:0; overflow:auto; }  /* fills */
.split > .pane:first-child { flex:0 0 auto; }  /* JS writes flex-basis here */

.split > .gutter { flex:0 0 10px; cursor:col-resize; touch-action:none; }
    
      
    3. 
      
  3. 
        
    3
    Pointer math: a drag position becomes a percent
    
        
    This is the whole trick. Take the pointer's coordinate, subtract the container's start edge, divide by the container's size, and you have "how far along, 0–100%". For a side-by-side split that is (clientX − left) / width; for a stacked split it is (clientY − top) / height. Using percent (not pixels) means the split keeps its proportions when the window resizes, and it is exactly the number aria-valuenow wants.
    
        
    function positionToPercent(split, clientX, clientY){
  const r = split.getBoundingClientRect();
  return split.classList.contains("is-row")
    ? (clientX - r.left) / r.width  * 100     // horizontal split
    : (clientY - r.top)  / r.height * 100;    // vertical split
}
    
      
    4. 
      
  4. 
        
    4
    Constraints: clamp, then write one style
    
        
    A raw percent can go negative or past 100 when the pointer leaves the container, so every value passes through one apply() gate that clamps it to [min, max] and does two things: sets the primary pane's flex-basis (the visual), and mirrors the same number into aria-valuenow (the semantic). Funnelling every change — drag, keyboard, reset — through this single function is what keeps the picture and the accessibility state impossible to desynchronise.
    
        
    const min = 20, max = 80;
let pos = 45;

function apply(p){
  pos = Math.min(max, Math.max(min, p));       // clamp to [min, max]
  primary.style.flexBasis = pos + "%";         // the ONLY style we write
  gutter.setAttribute("aria-valuenow", Math.round(pos));
  gutter.setAttribute("aria-valuetext", Math.round(pos) + "%");
}
    
      
    5. 
      
  5. 
        
    5
    One code path for mouse + touch: Pointer Events
    
        
    Rather than writing separate mouse and touch handlers, use Pointer Events, which unify mouse, touch and pen. On pointerdown, call setPointerCapture so every later move and up event is delivered to the gutter even when the pointer wanders off it — no document-level listeners, no "lost the drag" bugs. pointermove then just runs the math, and touch-action:none on the gutter stops the browser from scrolling instead of dragging.
    
        
    gutter.addEventListener("pointerdown", e => {
  gutter.setPointerCapture(e.pointerId);        // mouse OR touch OR pen
  split.classList.add("dragging");
  e.preventDefault();
});
gutter.addEventListener("pointermove", e => {
  if (!gutter.hasPointerCapture(e.pointerId)) return;   // only while dragging
  apply(positionToPercent(split, e.clientX, e.clientY));
});
gutter.addEventListener("pointerup", e => {
  gutter.releasePointerCapture(e.pointerId);
  split.classList.remove("dragging");
});
    
      
    6. 
      
  6. 
        
    6
    Double-click to reset
    
        
    People expect a divider to snap back to its default when double-clicked — it is the universal "undo my fiddling" gesture in editors and terminals. Because resetting is just "apply the default percent", it is a one-liner on top of the machinery you already have. Keep the default in a constant so the same value can seed the initial layout, the double-click, and the keyboard Enter reset.
    
        
    const DEFAULT = 45;

gutter.addEventListener("dblclick", () => apply(DEFAULT));   // snap back
apply(DEFAULT);                                              // seed on load
    
      
    7. 
      
  7. 
        
    7
    Orientation toggle: swap the axis + the ARIA
    
        
    Horizontal vs vertical is a one-class change: flip flex-direction between row and column and the same panes now stack. The only care point is ARIA — aria-orientation describes the separator line, so a side-by-side split has a vertical divider (aria-orientation="vertical") and a stacked split has a horizontal one. After flipping, re-apply the current percent so it is measured against the new axis.
    
        
    function setDir(row){
  split.classList.toggle("is-row", row);
  split.classList.toggle("is-col", !row);
  // the divider LINE is vertical when panes sit side-by-side:
  gutter.setAttribute("aria-orientation", row ? "vertical" : "horizontal");
  apply(pos);                                   // re-measure on the new axis
}
    
      
    8. 
      
  8. 
        
    8
    Keyboard: the WAI-ARIA window-splitter keys
    
        
    A mouse-only resizer is inaccessible, so the focused separator must move by keyboard. The splitter contract: arrow keys nudge it (Left/Up shrink the primary, Right/Down grow it), Shift takes bigger jumps, Home/End jump to min/max, and Enter resets. Every branch calls the very same apply(), so keyboard resizing is automatically clamped and mirrored into aria-valuenow just like dragging. Only preventDefault the keys you actually handle.
    
        
    gutter.addEventListener("keydown", e => {
  const step = e.shiftKey ? 10 : 2;             // Shift = bigger jumps
  switch (e.key){
    case "ArrowLeft": case "ArrowUp":    apply(pos - step); break;
    case "ArrowRight": case "ArrowDown": apply(pos + step); break;
    case "Home":  apply(min); break;            // collapse primary
    case "End":   apply(max); break;            // expand primary
    case "Enter": apply(DEFAULT); break;        // reset
    default: return;                            // let other keys through
  }
  e.preventDefault();
});

    9
    Generalise + nest: initSplit on every .split
    
        
    Wrap all of the above in one initSplit(splitEl) that reads its min/max/default from data attributes and scopes every query with :scope > so it only ever touches its own panes and gutter. Now nesting is free: put a second .split inside a pane, run initSplit on both, and the child resizes independently of the parent — no special case, no shared state. That single, self-contained instance is the whole widget.
    
        
    function initSplit(split){
  const primary = split.querySelector(':scope > .pane');
  const gutter  = split.querySelector(':scope > .gutter');
  const min = +split.dataset.min, max = +split.dataset.max;
  let pos = +split.dataset.pos;
  /* … apply(), pointer, dblclick, keydown, setDir — all scoped to `split` … */
}

document.querySelectorAll('.split').forEach(initSplit);   // outer + nested
