🎵 Spotify Playlist — The Track Table: A CSS-Grid List Where Every Row Shares One Column Template, the Index Cell Swaps Its Number for a ▶ Play Button on Hover While a Like Heart and ⋯ Menu Fade In, Clicking a Row Makes It the Now-Playing Track With a Green Title and Animated Equalizer Bars, a Sticky Column Header, Full Keyboard Row Navigation, and Album / Date Columns That Drop on a Phone

Hover a row and its number becomes a play button; click it and the track turns green with dancing equalizer bars — the track list at the heart of Spotify.

The track table is the busiest surface in Spotify, and it is pure CSS Grid: the column-label row and every song row share one grid-template-columns template, so #, title, album, date and duration all snap into perfectly aligned columns with no <table>. The signature move is the hover-swap: the index cell holds both the track number and a hidden ▶ play button stacked on top of each other, and a single :hover rule flips which one shows — meanwhile a like heart and a ⋯ menu fade in at the end of the row. Hover the rows to see it, then click any row (or the big green button) to make it the now-playing track: its title goes green and four equalizer bars animate in the index cell. The column header is sticky, arrow keys walk the list, and on a phone the album and date columns simply drop away. No library.

Open any playlist on Spotify, Apple Music or YouTube Music and you meet the same object: a big cover and title up top, then a grid of rows — number, song, album, date, length. It looks like a data table but there is no <table>; it is CSS Grid, one column template shared by the header and every row so everything aligns. 🔨 Build it yourself

    
A Spotify track list looks like a serious data table — five aligned columns, hover controls, a now-playing state with animated bars, a sticky header, keyboard support, a mobile fold — but there is no <table> anywhere. It is a stack of CSS-Grid rows that all share one grid-template-columns, so the header labels and every song snap into the same columns. The famous hover trick is just two elements in one cell — a number and a hidden ▶ button — with a single :hover rule choosing which shows. Now-playing is a class that turns the title green and reveals four bars on a staggered @keyframes loop. Everything else is small: position: sticky for the header, a roving tabindex for the arrow keys, and a media query that swaps the column template. No framework, no dependencies.

    
    
      
  1. 
        
    1
    The layout: a grid table, not a <table>
    
        
    The header row and every track row are display:grid with the same grid-template-columns. Because they share the template, column 3 of the header (Album) lines up perfectly with column 3 of each row — no <td>, no fixed widths guessed by hand. Storing that template in one CSS variable (--gtc) means the responsive fold later is a one-line change. Fractional units (fr) let Title take the lion's share while # stays a tight 16px.
    
        
    .pl { --gtc: 16px 4fr 2.6fr 2fr minmax(96px,auto); }  /* one source of truth */

.track-headrow,
.track-row {
  display:grid;
  grid-template-columns: var(--gtc);   /* header + rows share it -> aligned */
  gap:16px; align-items:center;
  padding:6px 12px; border-radius:6px;
}

    2
    Render rows from data
    
        
    Keep the songs as an array of objects and build the rows in JS. Each row gets a data-i index so any later handler knows which track it is, and a tabindex for keyboard focus. The index cell is where the magic will live, so it holds four children up front — the number, a ▶ button, an equalizer, and a ⏸ button — all but the number hidden by default.
    
        
    const TRACKS = [
  { title:"Midnight City Lights", artist:"Neon Skyline",
    album:"After Hours", added:"3 days ago", dur:"3:42" },
  /* … */
];
trackBody.innerHTML = TRACKS.map((t,i) => `
  <div class="track-row" role="row" data-i="${i}" tabindex="-1">
    <div class="col-index">
      <span class="t-num">${i+1}</span>
      <button class="t-play">▶</button>
      <span class="t-eq"><span></span><span></span><span></span><span></span></span>
      <button class="t-pause">⏸</button>
    </div>
    <div class="col-title">…</div> …
  </div>`).join("");

  3. 
        
    3
    The hover-swap: # ↔ ▶ in one cell
    
        
    This is the signature move. The number and the play button occupy the same index cell. By default the number shows and the button is display:none. One :hover rule on the row flips them — the number hides, the button appears — so it reads as the number turning into a ▶. It is pure CSS: no JavaScript touches this, and it is instant. (Scope it to :not(.playing) so the now-playing row keeps its equalizer instead.)
    
        
    .t-play { display:none; }               /* hidden until hover */

.track-row:not(.playing):hover .t-num  { display:none; }        /* number out */
.track-row:not(.playing):hover .t-play { display:inline-flex; } /* ▶ in       */

  4. 
        
    4
    The row's other hover reveals: heart + ⋯
    
        
    The like heart and the ⋯ menu sit in the last column but stay at opacity:0 until you hover the row — so a resting list is calm and only the row under the cursor shows its actions. Fading with opacity (not display) keeps them in the layout, so nothing shifts when they appear. A liked heart stays visible and turns green even after you move away, because "liked" is a persistent state, not a hover.
    
        
    .t-like, .t-more { opacity:0; transition:opacity .12s; }
.track-row:hover .t-like,
.track-row:hover .t-more { opacity:1; }     /* reveal on row hover */

.t-like.on { opacity:1; color:#1db954; }    /* liked = stays lit, green */

  5. 
        
    5
    Now playing: one class, many effects
    
        
    Clicking a row makes it the current track. In JS that is: strip .playing off whichever row had it, add it to the clicked one, remember the index. All the visuals are downstream CSS keyed off that single class — the title goes green, the number becomes an equalizer. State lives in one place (the class + a playingIndex number); the look is derived from it, so nothing can get out of sync.
    
        
    let playingIndex = -1;
function play(i){
  rows.forEach(r => r.classList.remove("playing","paused"));
  rows[i].classList.add("playing");
  playingIndex = i;
  render();                       // read-out follows the state
}
rows.forEach(r => r.addEventListener("click", () => play(+r.dataset.i)));

/* CSS reacts to the class */
.track-row.playing .t-title { color:#1db954; }   /* green now-playing title */

  6. 
        
    6
    The equalizer bars
    
        
    The dancing bars are four <span>s in a flex row, each animating its height on the same @keyframes loop but with a different negative animation-delay — that offset is what makes them look independent instead of marching in unison. They only render on the now-playing row, and hovering that row swaps the equalizer for a ⏸ pause button. Under prefers-reduced-motion the bars freeze, so the effect is honest but never nauseating.
    
        
    .t-eq { display:flex; align-items:flex-end; gap:2px; height:16px; }
.t-eq span { width:3px; background:#1db954; animation:eq .9s ease-in-out infinite; }
.t-eq span:nth-child(1){ animation-delay:-.20s; }   /* staggered starts */
.t-eq span:nth-child(2){ animation-delay:-.55s; }
.t-eq span:nth-child(3){ animation-delay:-.05s; }
.t-eq span:nth-child(4){ animation-delay:-.40s; }
@keyframes eq { 0%,100%{ height:25%; } 50%{ height:100%; } }

    7
    The sticky column header
    
        
    The row of column labels (#, Title, Album…) should stay put while the songs scroll under it. position:sticky; top:0 inside the scrolling playlist does exactly that with no JS and no scroll listener — the header scrolls normally until it hits the top, then pins. Give it a solid background and a z-index so tracks slide behind it cleanly instead of showing through.
    
        
    .pl { max-height:560px; overflow-y:auto; }   /* the scroll container */

.track-headrow {
  position:sticky; top:0; z-index:5;          /* pin to the top when scrolled */
  background:#181818;                          /* opaque so rows pass behind */
  border-bottom:1px solid rgba(255,255,255,.12);
}

  8. 
        
    8
    Keyboard row navigation
    
        
    A list you can only click is half-built. With a roving tabindex, exactly one row is focusable (tabindex="0") at a time; ↑/↓ move that focus to the neighbour, and Enter plays the focused track. Keeping one tab stop means Tab jumps into and out of the list in one press instead of trudging through every song — the pattern the ARIA grid guidelines recommend.
    
        
    let sel = 0;
function focusRow(i){
  sel = Math.max(0, Math.min(rows.length-1, i));
  rows.forEach((r,k) => r.tabIndex = k === sel ? 0 : -1);  // roving tabindex
  rows[sel].focus();
}
tracks.addEventListener("keydown", e => {
  if (e.key === "ArrowDown"){ e.preventDefault(); focusRow(sel+1); }
  if (e.key === "ArrowUp"){   e.preventDefault(); focusRow(sel-1); }
  if (e.key === "Enter" || e.key === " "){ e.preventDefault(); play(sel); }
});

    9
    Responsive: drop columns, and the a11y finish
    
        
    On a phone there is no room for five columns, so the fold is two moves: rewrite --gtc to just three tracks and display:none the Album and Date cells. Because every row reads the same variable, one rule reshapes the whole table. To finish it honestly, the table carries role="grid" with row/gridcell children and a visually-hidden aria-live region announces each change ("Playing Midnight City Lights.", "Added to Liked Songs.") so a screen-reader user hears exactly what a sighted user sees.
    
        
    @media (max-width:640px){
  .pl { --gtc: 16px 1fr auto; }              /* 5 columns -> 3 */
  .pl .col-album, .pl .col-date { display:none; }
}
/* announce every state change through one live region */
function announce(msg){ live.textContent = msg; }
play(i);  announce("Playing " + TRACKS[i].title + ".");
