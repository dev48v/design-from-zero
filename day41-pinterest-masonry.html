Pins of every height pack into tight columns with no gaps between rows — that is masonry, and there are two very different ways to get it.

A Pinterest board is a wall of pins of wildly different heights that still tile together with no ragged row gaps. Plain CSS Grid can't do that easily — its rows share a height, so a short pin next to a tall one leaves a hole. So you reach for one of two techniques. CSS columns is one line (column-count + break-inside:avoid) and packs beautifully, but it flows pins top-to-bottom down each column — so the reading order is column-major (watch the numbers). True masonry is a pinch of JavaScript: measure each pin and drop it into whichever column is currently shortest, which keeps the columns balanced and reads left-to-right. Flip the technique toggle to feel the difference, change the column count (or pick Auto and resize the window), turn on Show order # to see the reading-order caveat, and hover any pin to dim it and reveal its Save button and actions. 🔨 Build it yourself

    
Masonry is the Pinterest wall: pins of every height packed into columns with no row gaps. The catch is that CSS Grid can't do it cheaply — grid rows share a height, so ragged pins leave holes. You have two real options. CSS columns is a one-liner that packs top-to-bottom down each column (great, but the reading order is column-major). True masonry is a little JavaScript: measure each pin and drop it into the shortest column, which balances the wall and keeps DOM reading order. We'll build the pins, wire both techniques behind one toggle, add responsive columns and the hover Save overlay, and finish with accessibility. No framework, no dependencies.

    
    
      
  1. 
        
    1
    Why CSS Grid can't do true masonry
    
        
    In a normal CSS Grid, every cell in a row shares that row's track height — so a short pin beside a tall one gets stretched or leaves a gap, and the next row starts below the tallest item. Masonry needs each column to flow independently, pulling the next item up into the shortest gap. Grid's row model can't express that (the grid-template-rows: masonry proposal exists but is experimental and not shipped across browsers). That's the whole reason masonry is its own pattern — you either give up rows entirely (CSS columns) or place items yourself (JS).
    
        
    /* GRID tries to align rows -> ragged heights leave holes */
.grid { display:grid; grid-template-columns:repeat(4,1fr); gap:16px; }
/* a short pin next to a tall one: the row height = the TALL one,
   so the short pin's column has a gap before the next row starts.
   masonry wants the next item pulled UP into that gap -> grid can't. */

/* the experimental fix (NOT widely shipped yet): */
.grid { grid-template-rows: masonry; }   /* Firefox flag / draft spec */

    2
    Render the pins from data
    
        
    Keep the pins as an array of objects — a title, an author, a colour and, crucially, an aspect ratio that gives each pin a different height. Build the markup in JS. Each pin has an image block (its height comes from aspect-ratio so the widths can change without us re-doing the math), a hidden hover overlay, and a caption. A data-i index and an order badge let us show the reading order later.
    
        
    const PINS = [
  { title:"Cozy Reading Nook", author:"studio.lume",
    cat:"Decor", ratio:"3/4", grad:["#6366f1","#a855f7"] },
  /* … 18 pins, ratios from 16/9 (wide) to 9/16 (tall) … */
];
board.innerHTML = PINS.map((p,i) => `
  <div class="pin" data-i="${i}" role="listitem">
    <div class="pin-img"
         style="aspect-ratio:${p.ratio};
                background:linear-gradient(135deg,${p.grad[0]},${p.grad[1]})">
      <span class="pin-order">${i+1}</span>
      <div class="pin-overlay">…Save + actions…</div>
    </div>
    <div class="pin-cap">${p.title} · ${p.author}</div>
  </div>`).join("");

    3
    Technique A — CSS columns (one line)
    
        
    The cheapest masonry is column-count. The browser slices the container into N equal columns and pours the pins in. The one rule that makes it work is break-inside: avoid so a pin never splits across a column boundary. No JavaScript, no measuring — but the pins flow top-to-bottom down column 1, then column 2, so this is column-major (see step 4).
    
        
    .board { column-count:4; column-gap:16px; }   /* that's the whole layout */

.board .pin {
  break-inside:avoid;             /* never split a pin across columns */
  -webkit-column-break-inside:avoid;
  width:100%;                     /* fill the column */
  margin:0 0 16px;                /* the row gap */
}

    4
    The reading-order caveat of CSS columns
    
        
    This is the trade-off you must know. CSS columns lays pins out vertically: pin 1, 2, 3 go down the first column, then 4, 5, 6 down the second. So visually, pin 5 sits under pin 4, not to its right — the DOM order reads down-then-across, which is wrong for a feed where "newest first" should read left-to-right across the top. Turn on Show order # in the demo to see the numbers march downward. If order matters, you need technique B.
    
        
    /* CSS columns fills DOWN each column, then moves ACROSS:

   col 1   col 2   col 3   col 4
   [ 1 ]   [ 5 ]   [ 9 ]   [13 ]
   [ 2 ]   [ 6 ]   [10 ]   [14 ]
   [ 3 ]   [ 7 ]   [11 ]   [15 ]
   [ 4 ]   [ 8 ]   [12 ]   …

   reading order = column-major (down). For a "newest across
   the top" feed this is WRONG — that's why real feeds use JS. */

    Technique B — JS masonry: shortest-column placement
    
        
    True masonry places pins in DOM order, each into whichever column is currently shortest. Keep an array of running column heights, all starting at 0. For each pin, find the index of the minimum height, that's the pin's column; then add the pin's height (plus the gap) to that column's running total. Because you always feed the shortest column, the wall stays balanced and the pins read across-then-down in DOM order.
    
        
    const colH = new Array(cols).fill(0);   // running height of each column

pins.forEach(pin => {
  // 1) which column is shortest right now?
  let c = 0;
  for (let k = 1; k < cols; k++) if (colH[k] < colH[c]) c = k;

  // 2) place the pin there (step 6 does the pixel math)
  place(pin, c, colH[c]);

  // 3) grow that column by the pin's measured height + the gap
  colH[c] += pin.offsetHeight + GAP;
});

  6. 
        
    6
    Measure & absolutely position each pin
    
        
    JS masonry means you own the coordinates. Make the board position:relative and every pin position:absolute. Compute the column width from the container: (boardWidth − (cols−1)·gap) / cols, and set it on each pin first so aspect-ratio resolves the height. Then pin.offsetHeight gives the real measured height to feed the algorithm. The pin's left is its column index times (colW + gap); its top is that column's running height. Finally set the board's height to the tallest column so the page flows correctly.
    
        
    const boardW = board.clientWidth;
const colW = (boardW - GAP*(cols-1)) / cols;

pins.forEach(p => { p.style.position="absolute"; p.style.width = colW+"px"; });

const colH = new Array(cols).fill(0);
pins.forEach(p => {
  let c = 0; for (let k=1;k<cols;k++) if (colH[k] < colH[c]) c = k;
  p.style.left = c*(colW+GAP) + "px";     // horizontal slot
  p.style.top  = colH[c] + "px";          // stack under this column
  colH[c] += p.offsetHeight + GAP;        // measured height grows the column
});
board.style.height = Math.max(...colH) + "px";   // container must wrap them

  7. 
        
    7
    Responsive column count
    
        
    A board should show more columns on a wide screen and fewer on a phone. Derive the count from the container's width with a few breakpoints, and re-run the layout on resize (throttled with requestAnimationFrame so it stays smooth). For JS masonry you must re-layout on resize anyway, because the column width — and therefore every pin's height and position — changes. The pure-CSS route can be responsive too by swapping column-count for column-width, letting the browser fit as many columns as the width allows.
    
        
    function autoCols(w){ return w<480 ? 2 : w<720 ? 3 : w<1000 ? 4 : 5; }

let queued = false;
addEventListener("resize", () => {
  if (queued) return; queued = true;
  requestAnimationFrame(() => { queued = false; layout(); });  // re-place on resize
});

/* pure-CSS responsive alternative — no JS, browser fits the columns: */
.board { column-width:220px; column-gap:16px; }   /* as many as fit */

    8
    The hover overlay: dim + Save + actions
    
        
    Every pin carries an overlay that's invisible at rest (opacity:0) and fades in on hover. It's a dark gradient that dims the image so the controls read clearly: a red Save button pinned top-right, and a row of round action buttons at the bottom. Fading with opacity (not display) keeps the controls in the layout so nothing shifts. Save is a persistent toggle — a saved pin flips to a dark "Saved" state, which is state, not hover, so it stays saved after the cursor leaves.
    
        
    .pin-overlay {
  position:absolute; inset:0; opacity:0; transition:opacity .18s;
  background:linear-gradient(180deg, rgba(0,0,0,.32), transparent 40%,
                                     rgba(0,0,0,.46));      /* the dim */
  display:flex; flex-direction:column; padding:10px;
}
.pin:hover .pin-overlay,
.pin:focus-within .pin-overlay { opacity:1; }               /* reveal */

.pin-save   { align-self:flex-end; background:#e60023; color:#fff; }
.pin-save.saved { background:#111; }                        /* persistent state */
.pin-actions{ margin-top:auto; align-self:flex-end; }        /* bottom row */

    Wire it together + the a11y finish
    
        
    One layout() function reads the current mode and column count and calls either layoutCSS() (clear inline styles, set column-count) or layoutJS() (measure & place). Every control — the technique toggle, the column buttons, resize — just calls layout(), so state lives in two variables and the view is derived from them. Finish honestly: the board is a role="list" of listitem pins, each Save is a real <button> with an aria-pressed and aria-label, and a visually-hidden aria-live region announces each change ("Saved Cozy Reading Nook.", "Switched to JS masonry.") so a screen-reader user follows the same story.
    
        
    let mode = "css", colMode = 4;                 // the entire state
function resolveCols(){ return colMode==="auto" ? autoCols(board.clientWidth) : colMode; }
function layout(){
  const cols = resolveCols();
  mode === "css" ? layoutCSS(cols) : layoutJS(cols);   // one branch
  report(cols);                                        // read-out follows
}
/* announce every change through one live region */
function announce(msg){ live.textContent = msg; }
save.onclick = () => { const on = save.classList.toggle("saved");
  save.setAttribute("aria-pressed", on);
  announce((on?"Saved ":"Removed ") + PINS[i].title + "."); };
