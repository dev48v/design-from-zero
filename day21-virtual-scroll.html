📜 Virtual Scroll — 100,000 Rows, ~20 DOM Nodes

A list of 100,000 rows that scrolls at 60fps — with only ~20 rows in the DOM.

The box below holds one hundred thousand rows, but at any instant only the handful that fit the visible window actually exist as HTML. Scroll it and watch the live counter: the number of real DOM nodes stays tiny and flat no matter how far you fly down the list. A single empty spacer gives the scrollbar the full height so it behaves exactly as if all 100k rows were there. Then hit Render all (naive) to mount every row for real — feel the page hitch, watch the node count explode into six figures, and see why big lists need windowing. Toggle back to Virtual and it is instantly smooth again. All vanilla JS, fixed row height, no libraries.

Why the browser can't just render them all

Every DOM node costs memory, a slice of layout time, and paint work. A hundred thousand rows means the browser must create, style, lay out, and track a hundred thousand elements before a single pixel appears — seconds of frozen tab, hundreds of megabytes of memory, and janky scrolling forever after. Virtual scrolling flips it: the list data can be as large as you like, but the number of elements on screen is bounded by what fits the window. You render a moving window, not the whole document. That is the entire trick — and it is what powers every chat log, spreadsheet, and infinite feed you use.

Fixed height here · variable height is the advanced case

This demo uses a fixed row height, which makes the math exact: any row's top is just index × rowHeight, and the first visible row is Math.floor(scrollTop / rowHeight). When rows can be different heights you lose that closed-form answer and must measure rows as they render, keep a running offset map, and estimate the rest — libraries like TanStack Virtual and react-window exist largely to handle that bookkeeping. Start with fixed height, get the window right, then reach for a library the day you truly need variable rows.

The whole idea: a big list is just data. Keep the DOM the size of the window, give the scrollbar a spacer the size of the data, and move the window on scroll — that's virtual scrolling.

🔨 Build it yourself

    
A scrollable viewport, one tall empty spacer, and a fistful of absolutely-positioned rows. Compute which rows are visible from scrollTop, add a small overscan buffer, and re-render on scroll. Fixed row height keeps the math a one-liner. Pure CSS plus vanilla JS, no libraries — then a note on variable heights.

    
    

      
  1. 1
    The viewport + the tall empty spacer
    
        
    <!-- fixed-height scroll box; the sizer is empty but TALL -->
<div class="vs-viewport" id="viewport">
  <div class="vs-sizer" id="sizer"></div>
</div>

<style>
  .vs-viewport { position:relative; height:420px; overflow-y:auto; }
  .vs-sizer    { position:relative; width:100%; }  /* height set in JS */
  .vs-row      { position:absolute; left:0; right:0; height:36px; }
</style>

// give the scrollbar the FULL height of all rows, up front
const ROW_H = 36, TOTAL = 100000;
sizer.style.height = (ROW_H * TOTAL) + "px";   // 3,600,000px tall
    
      
    2. 

      
  2. 2
    Which rows are visible? (from scrollTop)
    
        
    function visibleRange() {
  const scrollTop = viewport.scrollTop;
  const height    = viewport.clientHeight;

  // fixed height ⇒ closed-form: first row = scrollTop / rowHeight
  const first = Math.floor(scrollTop / ROW_H);
  const count = Math.ceil(height / ROW_H);      // rows that fit the window
  const last  = Math.min(TOTAL - 1, first + count);
  return { first, last };
}
    
      
    3. 

      
  3. 3
    Add an overscan buffer
    
        
    const OVERSCAN = 4;   // render a few extra rows above & below

function windowRange() {
  const { first, last } = visibleRange();
  return {
    from: Math.max(0, first - OVERSCAN),
    to:   Math.min(TOTAL - 1, last + OVERSCAN),
  };
}
// the buffer means rows already exist just off-screen,
// so fast scrolling never flashes a blank gap
    
      
    4. 

      
  4. 4
    Render only that window, positioned by index
    
        
    function render() {
  const { from, to } = windowRange();
  sizer.innerHTML = "";                     // clear the old window
  const frag = document.createDocumentFragment();
  for (let i = from; i <= to; i++) {
    const row = document.createElement("div");
    row.className = "vs-row";
    row.style.top = (i * ROW_H) + "px";     // absolute position by index
    row.textContent = "Row " + i;
    frag.appendChild(row);
  }
  sizer.appendChild(frag);
}

  5. 5
    Re-render on scroll (throttled with rAF)
    
        
    let ticking = false;
viewport.addEventListener("scroll", () => {
  if (ticking) return;
  ticking = true;
  // coalesce many scroll events into one paint per frame
  requestAnimationFrame(() => { render(); ticking = false; });
});
render();   // first paint

  6. 6
    Skip re-render when the window hasn't changed
    
        
    let lastFrom = -1;

function render() {
  const { from, to } = windowRange();
  if (from === lastFrom) return;   // still the same window → do nothing
  lastFrom = from;
  /* …build rows from..to as before… */
}
// most scroll pixels don't cross a row boundary,
// so this skips the vast majority of DOM work

    Optional: recycle nodes instead of rebuilding
    
        
    // advanced: keep a pool of ~(window+overscan) row elements and just
// re-point each one's top + text as the window moves. No create/destroy.
pool.forEach((row, k) => {
  const i = from + k;
  row.style.top   = (i * ROW_H) + "px";
  row.textContent = "Row " + i;
  row.style.display = (i <= to) ? "" : "none";
});
// fewer allocations, less GC → smoother on low-end devices
