🐦 X / Twitter Feed — A Reverse-Chronological Timeline: Seeded Posts Newest-First, a Compose Box That Prepends a New Post Behind a 280-Character Ring, Optimistic Like / Repost / Reply Counts That Toggle Instantly, and Relative Timestamps Derived From One Number

Write a post and watch it jump to the top. Like it, repost it, reply — the counts flip the instant you click.

A working X / Twitter feed built from scratch in vanilla JS — the timeline that every social app copies. One idea makes it work: the feed is just an array of post objects, and what you see on screen is a pure function of that array. Sort it newest-first and it is a reverse-chronological timeline; unshift a new post and it appears at the top; flip a liked boolean and nudge its count by one and the heart fills optimistically — no round-trip, no reload. Even the "5m" and "2h" timestamps are just arithmetic on one number: now − posted-at. Type in the box (the 280 ring fills, then turns amber, then red), hit Post or ⌘/Ctrl+Enter, and tap the action buttons. No library.

The screen is a render(state). Change the array — post, like, repost — and re-draw. Nothing on screen is edited in place.

Every timeline you scroll — X, Bluesky, Mastodon, a comments thread, a chat log, an activity feed — is this same shape: a list of records held in memory, sorted by time, rendered to a column of rows, with each row's little counters and its "3m ago" label computed from the record rather than stored as text. Interactions are optimistic — the UI trusts the click and updates instantly, reconciling with the server later — which is why liking feels weightless. The whole thing is a dozen lines: an array, a sort, a template, and one function that flips a flag and re-draws.

🔨 Build it yourself

    
A feed looks like a lot of moving parts — timestamps ticking, hearts filling, new posts sliding in — but underneath it is one array of plain objects and a single render() that draws whatever the array currently says. Keep the posts sorted newest-first and you have a reverse-chronological timeline; unshift a new object and it lands at the top; flip a liked boolean and adjust its count and re-draw, and the like registers optimistically with zero latency. Relative time is not stored, it is derived — now − postedAt turned into "now / 5m / 2h / 1d". The compose box counts toward 280 and disables Post when empty or over. One toggle() serves like, repost and reply; buttons and the keyboard share it; and labelled buttons plus a live region make it usable by everyone. ~120 lines of vanilla JS, no dependencies.

    The markup: a compose box + an empty feed
    
        
    Two regions and nothing more: a .compose box with a growing <textarea>, a character ring and a Post button; and an empty #feed column that JS fills. The whole thing is a labelled role="region", and a visually-hidden aria-live region will speak each action ("Posted.", "Liked …") to a screen reader. Note there is no post markup here at all — every row is generated from the data.
    
        
    <div class="feed" role="region" aria-label="Home timeline">
  <div class="compose">
    <div class="avatar">YO</div>
    <textarea id="composer" placeholder="What's happening?"></textarea>
    <svg id="ring">…character ring…</svg>
    <button id="postBtn" disabled>Post</button>
  </div>
  <div id="feed"><!-- posts injected here --></div>
</div>
<div class="sr-only" role="status" aria-live="polite"></div>

    2
    The data model: one post object
    
        
    Everything the UI needs lives in a plain object. The identity fields (name, handle, avatar) render the header; text is the body; time is a millisecond timestamp that will become the relative label; and each action carries both a count and a boolean — likes/liked, reposts/reposted, replies/replied — because the button needs to show a number and whether you have toggled it. The id lets a click find its post again.
    
        
    let uid = 1;
const posts = [
  { id: uid++, name:"Sarah Drasner", handle:"@sarah_edo", ini:"SD",
    grad:"linear-gradient(135deg,#6366f1,#a855f7)",
    text:"a timeline is just a sorted array.",
    time: Date.now() - 8_000,        // 8 seconds ago
    replies:12,  replied:false,
    reposts:48,  reposted:false,
    likes:302,   liked:false },
  // …more, oldest last
];

    3
    Relative time: one number → "now / 5m / 2h"
    
        
    A feed never stores "3 minutes ago" as text — it stores a timestamp and derives the label every time it draws, so it is always correct even minutes later. Subtract the post's time from Date.now(), then walk the units: under 5s is "now", under a minute is seconds, then minutes, then hours, then days, and past a week you fall back to an actual date. Keep the real timestamp in a title so hovering shows the exact moment.
    
        
    function relTime(ts){
  const s = Math.max(0, Math.floor((Date.now() - ts) / 1000));
  if (s < 5)   return "now";
  if (s < 60)  return s + "s";
  const m = Math.floor(s / 60); if (m < 60) return m + "m";
  const h = Math.floor(m / 60); if (h < 24) return h + "h";
  const d = Math.floor(h / 24); if (d < 7)  return d + "d";
  return new Date(ts).toLocaleDateString(undefined,{month:"short",day:"numeric"});
}

  4. 
        
    4
    Render the feed, newest-first
    
        
    This is the heart of it: render is a pure function of the array. Sort a copy by time descending so the newest post is first — that single line is "reverse-chronological" — then map each post to a row of HTML and drop it into #feed. Because we redraw the whole list from state, there is never any stale DOM to reconcile by hand: change the data, call render, done. User text is escaped (step 5) before it touches innerHTML.
    
        
    const feed = document.getElementById("feed");

function renderFeed(){
  const ordered = [...posts].sort((a, b) => b.time - a.time); // newest first
  feed.innerHTML = ordered.map(postHTML).join("");
  updateReadout();
}

function postHTML(p){
  return `<article class="post" data-id="${p.id}">
    <div class="avatar" style="background:${p.grad}">${p.ini}</div>
    <div class="post-main">
      <div class="post-head">
        <span class="post-name">${esc(p.name)}</span>
        <span class="post-handle">${esc(p.handle)}</span> <span>·</span>
        <span class="post-time" data-time="${p.time}">${relTime(p.time)}</span>
      </div>
      <div class="post-text">${esc(p.text)}</div>
      ${actionsHTML(p)}
    </div>
  </article>`;
}

  5. 
        
    5
    Escape the text — the post is user input
    
        
    The compose box lets anyone type <script> or &, and we build rows with innerHTML, so every user-supplied string must be escaped first or the feed becomes an XSS hole. A tiny replacer swaps the five HTML-significant characters for entities. It costs one function and it is non-negotiable the moment real people can post — a lesson every social product learns early.
    
        
    function esc(s){
  return s.replace(/[&<>"']/g, c => ({
    "&":"&amp;", "<":"&lt;", ">":"&gt;",
    '"':"&quot;", "'":"&#39;"
  })[c]);
}
// esc("<b>hi</b> & bye")  →  "&lt;b&gt;hi&lt;/b&gt; &amp; bye"

  6. 
        
    6
    The 280 counter: a ring that goes amber then red
    
        
    On every input, read the length, compute remaining = 280 − length, and drive an SVG progress ring by length / 280 via stroke-dashoffset. Colour is a signal: blue while there is room, amber in the last 20 characters, red once you are over. Crucially we do not hard-cap the textarea — letting people run over (Post disabled, count negative) tells them exactly how much to trim, which feels better than a wall.
    
        
    const MAX = 280, C = 2 * Math.PI * 11;   // ring circumference
function updateComposer(){
  const len = composer.value.length, remaining = MAX - len, over = len > MAX;
  ringProg.style.strokeDashoffset = C * (1 - Math.min(len, MAX) / MAX);
  ringProg.style.stroke = over ? "#f4212e" : remaining <= 20 ? "#f59e0b" : "#1d9bf0";
  ringNum.textContent  = remaining <= 20 ? remaining : "";   // show only near limit
  postBtn.disabled = len === 0 || over;                      // empty or over → blocked
  updateReadout();
}
composer.addEventListener("input", updateComposer);

    7
    Post: prepend a new object, re-draw
    
        
    Publishing is pure data work: build a fresh post object stamped with Date.now() and zeroed counters, then unshift it onto the array. Because render sorts by time, the new post — being the newest — lands at the very top; the .post fade-in animation makes it slide in. Clear the box, reset the counter, re-render, and announce it. That is the entire "reverse-chronological insert": add the newest item, sort, draw.
    
        
    function publish(){
  const text = composer.value.trim();
  if (!text || text.length > MAX) return;
  posts.unshift({                       // newest → front of the array
    id: uid++, name:"You", handle:"@you", ini:"YO", grad:MY_GRAD,
    text, time: Date.now(),
    replies:0, replied:false, reposts:0, reposted:false, likes:0, liked:false
  });
  composer.value = ""; updateComposer();
  renderFeed();
  announce("Posted. " + posts.length + " posts in the timeline.");
  composer.focus();
}

  8. 
        
    8
    Optimistic toggle: flip a flag, nudge a count
    
        
    Liking should feel free, so we update the model immediately and trust it — that is what "optimistic" means. Find the post by id, flip its boolean, and move its count by exactly ±1 in the same direction; re-render and the heart fills or empties with the new number. A real app would fire a request in the background and only roll this back if the server rejected it, but the instant local update is what makes the interaction feel weightless.
    
        
    const FIELD = { like:["liked","likes"], repost:["reposted","reposts"], reply:["replied","replies"] };

function toggle(id, kind){
  const p = posts.find(x => x.id === id); if (!p) return;
  const [flag, count] = FIELD[kind];
  p[flag]   = !p[flag];                 // flip the boolean
  p[count] += p[flag] ? 1 : -1;         // optimistic ±1
  renderFeed();
  announce((p[flag] ? "Toggled on " : "Toggled off ") + kind + " on " + p.name + "'s post.");
}

    9
    One click handler + the keyboard, same path
    
        
    Rather than wire a listener onto every button in every row — which breaks the moment you re-render — attach one click listener to #feed and let it delegate: find the clicked .act, read its data-act and the row's id, and call the shared toggle(). The compose Post button and a ⌘/Ctrl+Enter shortcut both call publish(). Every input funnels through one of two functions, so behaviour, animation and announcements are automatic and never duplicated.
    
        
    feed.addEventListener("click", e => {
  const btn = e.target.closest(".act"); if (!btn) return;
  const id = +btn.closest(".post").dataset.id;
  toggle(id, btn.dataset.act);          // one handler for all rows + actions
});

postBtn.addEventListener("click", publish);
composer.addEventListener("keydown", e => {
  if ((e.metaKey || e.ctrlKey) && e.key === "Enter"){ e.preventDefault(); publish(); }
});

    Accessibility + keeping the clock live
    
        
    Each action button gets a spoken label and an aria-pressed that mirrors its boolean, so a screen-reader user hears "Like, pressed" the moment you toggle it; every post and toggle also writes a sentence to the aria-live region. Finally, relative time drifts, so a small setInterval re-derives every visible "Nm" label every 10 seconds without touching the rest of the DOM — cheap, and it keeps "now" from lying. That is the whole feed.
    
        
    function actionsHTML(p){
  return `<div class="post-actions">
    <button class="act like ${p.liked?'on':''}" data-act="like"
            aria-pressed="${p.liked}" aria-label="Like, ${p.likes} likes">
      ${heart(p.liked)}<span>${p.likes}</span>
    </button> …repost, reply… </div>`;
}
// keep timestamps honest
setInterval(() => {
  feed.querySelectorAll(".post-time").forEach(el =>
    el.textContent = relTime(+el.dataset.time));
}, 10_000);
function announce(msg){ live.textContent = msg; }   // aria-live region
