💬 Smart Tooltip — Flip & Shift to Stay On-Screen

Hover the buttons — especially the ones jammed into the corners.

A naïve tooltip always opens above its trigger. But a trigger near the top edge would push the tooltip off-screen, and one near the right edge would clip it. A smart tooltip measures itself and the viewport, then flips to the opposite side when it would overflow and shifts sideways to stay fully visible — the core trick behind libraries like Floating UI, hand-built here in vanilla JS. Hover or keyboard-focus (Tab) any button; watch the readout show the placement it chose.

Chosen placement: —
Shift: 0px

Preferred side is top. When the tooltip would overflow the top it flips to bottom (and vice-versa on the sides); then it's clamped horizontally so it never leaves the viewport, and the little arrow slides to stay pointed at the trigger. Everything is measured live with getBoundingClientRect() — resize the window and re-hover to see the decisions change.

🔨 Build it yourself

    
One shared floating element, a measure step with getBoundingClientRect(), a flip decision, a shift-and-clamp, an arrow that tracks the trigger, and hover/focus wiring with a small delay. ~80 lines of vanilla JS.

    
    
      
  1. 
        
    1
    One reusable floating element
    
        
    Instead of a tooltip per trigger, keep a single #tip element positioned with position:fixed so its coordinates are relative to the viewport (not any scrolled parent). It stays hidden until a trigger asks for it, then we move and fill it. pointer-events:none means the tooltip never steals the hover from the element underneath.
    
        
    const tip = document.getElementById("tip");   // position:fixed; opacity:0
function showTip(target, text){
  tip.querySelector("#tipText").textContent = text;
  tip.classList.add("show");
  place(target);        // measure + position (next steps)
}
function hideTip(){ tip.classList.remove("show"); }
    
      
    2. 
      
  2. 
        
    2
    Measure the trigger & the tooltip
    
        
    getBoundingClientRect() gives the trigger's position and size in viewport coordinates, and the tooltip's own width/height once it's rendered. With those two rectangles and the viewport size (innerWidth/innerHeight) we have everything needed to decide where it fits. Measuring the tooltip after making it visible is important — a hidden element has no size.
    
        
    const r = target.getBoundingClientRect();   // trigger box
const t = tip.getBoundingClientRect();      // tooltip box
const vw = innerWidth, vh = innerHeight;
const GAP = 8;   // space between trigger and tooltip
    
      
    3. 
      
  3. 
        
    3
    Flip when the preferred side overflows
    
        
    We prefer top. If placing the tooltip above the trigger would push its top edge above 0 (off the top of the screen), we flip to bottom. The vertical coordinate is computed for whichever side we settle on. The same idea handles left/right for side placements — always flip to the side with more room.
    
        
    let placement = "top";
let top = r.top - t.height - GAP;      // try above
if (top < 0){ placement = "bottom"; top = r.bottom + GAP; }
// (if bottom also overflowed we'd clamp; here top/bottom always fit)
    
      
    4. 
      
  4. 
        
    4
    Centre, then shift & clamp
    
        
    Horizontally we start centred on the trigger, then clamp so the tooltip never crosses the left (0) or right (vw − width) edge. The amount we moved from centre is the "shift". We remember it so the arrow can be nudged back the other way and keep pointing at the trigger.
    
        
    let left = r.left + r.width/2 - t.width/2;   // centred
const clamped = Math.max(GAP, Math.min(left, vw - t.width - GAP));
const shift = left - clamped;                // how far we moved
left = clamped;
tip.style.left = left + "px";
tip.style.top  = top  + "px";
    
      
    5. 
      
  5. 
        
    5
    Arrow, delay, keyboard & ARIA
    
        
    The arrow sits at the tooltip's edge, offset by the shift so it still points at the trigger's centre. A short show-delay (~80ms) avoids flicker when the pointer just passes over. We wire mouseenter/leave and focus/blur so keyboard users get tooltips too, add role="tooltip" + aria-describedby for screen readers, and close on Esc.
    
        
    arrow.style.left = (t.width/2 + shift - 5) + "px";
arrow.style.top  = placement === "top" ? "100%" : "-10px";

trigger.addEventListener("mouseenter", () => timer = setTimeout(open, 80));
trigger.addEventListener("mouseleave", () => { clearTimeout(timer); hideTip(); });
trigger.addEventListener("focus", open);
trigger.addEventListener("blur", hideTip);
addEventListener("keydown", e => { if (e.key === "Escape") hideTip(); });
