🌲 Tree View — A Recursive File Explorer: Expand/Collapse, Roving-Tabindex Keyboard Nav & Full ARIA tree Roles

Expand a folder, collapse it, arrow your way around.

A working file-explorer tree built from scratch in vanilla JS — the same widget you use in VS Code's sidebar, a file dialog, or a nested-comment thread. Two ideas make it work: it is drawn by a recursive render (a folder renders its label, then renders each child the exact same way), and it is driven by one tab stop that roams — a roving tabindex — so the whole tree behaves like a single control. Click a chevron or a folder to open/close it. Then click the tree and try the keyboard: ↑ ↓ move, → open / dive in, ← close / jump to parent, Enter select, Home/End ends, or just type a letter to jump. No library.

Every tree you have ever used — a file sidebar, a category picker, a threaded comment list, an org chart, a JSON viewer — is this same shape: a recursive list of nodes, each folder holding a collapsible role="group" of children. The visible rows are whatever survives after you drop every collapsed subtree, and the keyboard treats that visible list as one roving-focus control. That is the whole widget.

🔨 Build it yourself

    
A tree view is a recursive data structure wearing a recursive UI. The data is an array of nodes, where a folder node carries its own array of children. You render it with a function that draws one node and then calls itself for each child, wrapping them in a collapsible role="group". Expand/collapse is just toggling one class plus aria-expanded. The keyboard is the interesting part: you compute the list of currently visible rows and move a single roving tabindex=0 through it. ~110 lines of vanilla JS, no dependencies.

    
    
      
  1. 
        
    1
    The data: recursive nodes
    
        
    A tree is a nested list. Each node has a name and a type; a folder additionally carries a children array (which may hold more folders, nesting as deep as you like), and an open flag for its starting state. This one recursive shape is what lets the same rendering and navigation code handle two levels or twenty.
    
        
    const TREE = [
  { name:"src", type:"folder", open:true, children:[
    { name:"components", type:"folder", children:[
      { name:"Button.jsx", type:"file" },
      { name:"TreeView.jsx", type:"file" },
    ]},
    { name:"index.js", type:"file" },
  ]},
  { name:"package.json", type:"file" },
];

    2
    Recursive render: a node draws its children the same way
    
        
    buildNode makes one role="treeitem" containing a label row, and — if it is a folder — a role="group" into which it appends buildNode(child, level+1) for every child. The function calling itself is the entire trick: depth is just how many times the recursion nested. Pass the level down so each item can set aria-level and indent its row.
    
        
    function buildNode(node, level){
  const item = el("div", { role:"treeitem", "aria-level":level });
  item._node = node;
  item.tabIndex = -1;
  item.appendChild(buildRow(node, level));      // chevron + icon + label
  if (node.type === "folder"){
    item.setAttribute("aria-expanded", !!node.open);
    const group = el("div", { role:"group", class:"tv-group" });
    node.children.forEach(c => group.appendChild(buildNode(c, level+1))); // ↻ recurse
    group.classList.toggle("open", !!node.open);
    item.appendChild(group);
  }
  return item;
}

    3
    Expand / collapse = one class + one attribute
    
        
    Opening or closing a folder does not touch the DOM structure — the children are always built and always present. You just toggle the child group's .open class (CSS shows/hides it), flip aria-expanded so assistive tech knows the state, and rotate the chevron. Because the subtree stays in the DOM, its state is preserved when you re-open it.
    
        
    function setOpen(item, open){
  if (item._node.type !== "folder") return;
  item.setAttribute("aria-expanded", open);
  item.querySelector(":scope > .tv-group").classList.toggle("open", open);
  item.querySelector(":scope > .tv-row .tv-chev").classList.toggle("open", open);
  item._node.open = open;
}
const toggle = item => setOpen(item, item.getAttribute("aria-expanded") !== "true");

    4
    The visible-nodes list — the backbone of navigation
    
        
    The keyboard does not care about the tree's shape; it cares about the flat list of rows the user can actually see right now. A node is visible only if every folder above it is open — so walk up from each item and drop it if any ancestor group is collapsed. ↑/↓ then become "previous / next in this list", regardless of nesting depth.
    
        
    const allItems = () => [...tree.querySelectorAll('[role="treeitem"]')];

function visibleItems(){
  return allItems().filter(item => {
    for (let g = item.parentElement; g && g !== tree; g = g.parentElement)
      if (g.classList.contains("tv-group") && !g.classList.contains("open"))
        return false;                 // an ancestor is collapsed → hidden
    return true;
  });
}

  5. 
        
    5
    Roving tabindex: the tree is ONE tab stop
    
        
    A tree with 50 files must not be 50 Tab stops. So exactly one item holds tabindex=0 and every other holds tabindex=-1. To move focus you set them all to -1, give the target 0, and call .focus(). Tab now enters and leaves the whole tree in one hop, and the arrow keys do the moving inside.
    
        
    let current = null;                    // the item that currently owns focus

function focusItem(item){
  if (!item) return;
  allItems().forEach(it => it.tabIndex = -1);   // everyone out of tab order
  item.tabIndex = 0;                            // …except the current one
  item.focus();
  current = item;
}

  6. 
        
    6
    The keyboard model (WAI-ARIA tree keys)
    
        
    This is the standard tree contract. Up/Down step through the visible list. Right is context-sensitive: on a closed folder it opens it, on an already-open folder it dives to the first child, on a file it does nothing. Left mirrors it: on an open folder it closes, otherwise it jumps to the parent folder. Home/End go to the first/last visible row.
    
        
    tree.addEventListener("keydown", e => {
  const item = e.target.closest('[role="treeitem"]'); if (!item) return;
  const vis = visibleItems(), i = vis.indexOf(item);
  const folder = item._node.type === "folder";
  const open = item.getAttribute("aria-expanded") === "true";
  switch (e.key){
    case "ArrowDown": focusItem(vis[i+1]); break;
    case "ArrowUp":   focusItem(vis[i-1]); break;
    case "ArrowRight":
      if (folder && !open) setOpen(item, true);
      else if (folder)     focusItem(firstChild(item));
      break;
    case "ArrowLeft":
      if (folder && open)  setOpen(item, false);
      else                 focusItem(parentItem(item));
      break;
    case "Home": focusItem(vis[0]); break;
    case "End":  focusItem(vis[vis.length-1]); break;
    case "Enter": case " ": if (folder) toggle(item); select(item); break;
    default: return;                 // let other keys (type-ahead) fall through
  }
  e.preventDefault();
});

    7
    Move within the tree: first child & parent
    
        
    Right-into and Left-to-parent need two tiny DOM lookups. The first child is the first treeitem inside this item's own group. The parent is found by climbing out of the containing group with closest — which returns null at the root, so top-level Left simply does nothing. Both use :scope so they only match direct descendants, never grandchildren.
    
        
    const firstChild = item =>
  item.querySelector(':scope > .tv-group > [role="treeitem"]');

const parentItem = item =>
  item.parentElement.closest('[role="treeitem"]');   // null at the root

  8. 
        
    8
    ARIA: tree · treeitem · group, plus selection
    
        
    The roles form a contract screen readers understand: role="tree" on the container, role="treeitem" on every node, and role="group" on each folder's child list. Folders expose aria-expanded; every item exposes aria-level (depth, 1-based) so users hear "level 3". Selection (distinct from focus) is carried by aria-selected. Give each item an explicit aria-label so its name is not polluted by the child text nested inside it, and mirror moves into a polite live region.
    
        
    <div role="tree" aria-label="File explorer">
  <div role="treeitem" aria-level="1" aria-expanded="true" aria-label="src" tabindex="0">
    <div class="tv-row">…chevron · icon · “src”…</div>
    <div role="group">
      <div role="treeitem" aria-level="2" aria-label="index.js" tabindex="-1">…</div>
    </div>
  </div>
</div>

function select(item){
  allItems().forEach(it => it.setAttribute("aria-selected", it === item));
  live.textContent = item._node.name + ", " +
    (item._node.type === "folder" ? "folder" : "file");
}

    9
    Polish: type-ahead, expand-all, indentation
    
        
    A few touches make it feel native. Type-ahead: pressing a printable key jumps to the next visible node whose name starts with it, wrapping around — invaluable in a long list. Expand-all / collapse-all loop over every folder and set the state. And indentation comes purely from aria-level: pad each row by level × 16px so depth is visible, never hard-coded per node.
    
        
    function typeahead(ch){
  const vis = visibleItems(), start = vis.indexOf(current);
  for (let k = 1; k <= vis.length; k++){
    const it = vis[(start + k) % vis.length];      // search forward, wrap
    if (it._node.name.toLowerCase().startsWith(ch.toLowerCase()))
      return focusItem(it);
  }
}
// indentation is a pure function of depth:
row.style.paddingLeft = (level * 16) + "px";
